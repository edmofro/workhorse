Same patternIdentical input box structure in both contexts — textarea, hairline, scrollable action bar, pinned right button. The right button changes meaning: "Create card" in the modal, "Send" in the card. Same shape, different colour (accent vs near-black).
Skill sourceIn creation, skills are fixed (the full set). In the card, skills are jockey-generated — so the scrollable track contents change over the card's lifecycle. Early: Workshop, Interview, Draft spec. Mid-implementation: Continue, Update spec, Code review. The scroll behaviour handles any number gracefully.
No kbd hints in-card⌘1–⌘4 shortcuts omitted from in-card skills — the set is dynamic so the shortcuts can't be stable. Could add them if we decide the jockey always emits skills in a consistent order, but that's a separate decision.
Property stripCreation uses the property name ("Priority", "Assignee"). In-card uses action form ("Set priority", "Assign") — more explicit about what clicking does since you're already on the card and editing metadata feels more deliberate.
