The AI generates HTML/CSS mockups during the chat conversation. Mockups are artifacts — they open in the same chat + artifact layout as specs (see `card-navigation.md`). Clicking a mockup opens it alongside the chat, not as a full-screen overlay. There may be many mockups over the course of specifying a card: a date picker detail, a full page layout, a merge conflict dialog, etc.

## Context

"The tricky bit is that we generally refine things with a reference point of a visual design in Figma, and we understand the specs more and more as we develop it and see it coming to life. That's a harder process with just words."

## How mockups are created

Mockups come from the chat — either proactively generated by the AI or requested by the user.

- [ ] The AI **proactively generates** mockups during interviews and spec drafting whenever a visual would help illustrate the concept being discussed, without waiting to be asked
- [ ] The AI asks "would a mockup help here?" when discussing UI-heavy features
- [ ] The user can request a mockup explicitly in the chat
- [ ] The **"Add a mockup"** action pill is available on card home to kick off mockup creation — user pastes Figma HTML, describes what they want, or attaches a screenshot
- [ ] Pasting raw HTML (e.g. from Figma) into the chat is the way to provide material for the AI to clean up into a mockup — the editor is for tweaking results, not for initial creation

## In the chat

- [ ] Mockups appear as compact preview cards in the conversation where they were generated
- [ ] Each mockup has a title describing what it shows (e.g., "Patient allergies — list view")
- [ ] Multiple mockups accumulate over the course of a conversation
- [ ] Clicking a mockup preview card opens it as an artifact (chat slides left, mockup opens right)

## Files panel

Mockups are listed alongside specs in the files panel (see `card-navigation.md` for panel behaviour).

- [ ] Each entry shows the mockup title (derived from HTML `<title>` element, see `labels.md`)
- [ ] Clicking an entry opens the mockup as an artifact
- [ ] The panel is available in every view state (card home, chat, artifact)

## Mockup artifact (viewing)

Mockups open in the same artifact layout as specs: chat on the left (~40%), mockup on the right (~60%). The artifact header bar has the same chrome as specs, plus mockup-specific additions.

- [ ] Rendered HTML preview, read-only by default
- [ ] Subtle border/background to distinguish the mockup from the app itself
- [ ] **Device toggle** in the header bar: Desktop, Tablet, Mobile — switches the preview's aspect ratio and width constraints
- [ ] ◀ ▶ arrows flip between this card's files (specs and mockups intermixed)
- [ ] ⌄ dropdown for file browser with search
- [ ] ✕ close returns to centred chat
- [ ] **Expand icon** in the header bar collapses the chat column to a single icon on the left edge, giving the mockup full width. Click the icon to restore the chat column. Primarily useful for viewing mockups at desktop aspect ratios on smaller screens

## Mockup artifact (editing)

Editing a mockup is a stopgap for when the AI keeps getting a specific thing wrong and the user wants to just fix it themselves. It's not a design tool — it's a quick-fix escape hatch.

### Split view

- [ ] Clicking **Edit** in the header bar enters edit mode: the artifact area splits into **preview on top, editor panel on bottom**
- [ ] The preview updates live as the user makes changes in the editor panel
- [ ] Chat remains visible on the left — no layout change beyond the split within the artifact area
- [ ] **"Done editing"** collapses the editor panel, returns to preview-only, and triggers auto-commit

### Editor panel tabs

The bottom editor panel has two tabs:

**Properties tab:**
- [ ] When an element is selected in the preview (click to select), shows its editable CSS properties as form fields
- [ ] Common properties exposed as labelled inputs: padding (top/right/bottom/left), margin, border-radius, background colour (with colour picker), text colour, font-size, font-weight, text content
- [ ] The user doesn't need to know CSS syntax — just type "16" in the padding field or pick a colour
- [ ] Changes apply immediately and update the preview
- [ ] Covers ~80% of quick-fix tweaks (spacing, colours, typography, rounding)

**Source tab:**
- [ ] Raw HTML source of the full mockup in a code editor
- [ ] **Click-to-focus:** clicking an element in the preview while the Source tab is active scrolls the source editor to that element's code and highlights it
- [ ] For power users who need more than the Properties tab offers
- [ ] For pasting in HTML that the AI should then clean up (though pasting into chat is preferred for that)
- [ ] Changes in the source update the preview live

### Element selection in preview

- [ ] In edit mode, clicking an element in the preview selects it (visual highlight/outline)
- [ ] The Properties tab populates with that element's current styles
- [ ] The Source tab scrolls to and highlights that element's HTML
- [ ] Clicking a different element changes the selection
- [ ] Clicking empty space deselects

## Mockup format

- [ ] Rendered inline, not images
- [ ] The AI generates them the same way a developer would mock up a UI (HTML/CSS)
- [ ] Each mockup is a standalone HTML file with inline CSS
- [ ] Mockups reference the project's design system palette for consistency

## Committing and storage

Mockups are saved to `.workhorse/design/mockups/{card-id}/` and auto-committed alongside specs (see `commit-specs.md`).

- [ ] Each mockup includes an HTML comment header referencing the spec(s) it represents:

```html
<!--
  spec: patient/allergies.md
  card: WH-042
  title: Allergy list — main view
-->
```

- [ ] Mockups auto-commit when the user leaves edit mode ("Done editing") or when the agent finishes a turn
- [ ] WIP mockups being iterated on in chat are committed on each agent turn

## Reference direction

- [ ] Specs do NOT reference mockups — mockups may be temporary and deleted after implementation
- [ ] Mockups reference specs via their HTML comment header, so the AI can search for relevant mockups when implementing a given spec

## As implementation context

- [ ] Mockup HTML is included in the downloadable implementation prompt
- [ ] AI agents search for relevant mockups by spec path when implementing
- [ ] Mockups can be deleted without breaking any spec references

## Open questions

> **Properties tab scope:** Which CSS properties should the Properties tab expose? Starting with padding, margin, border-radius, background-color, color, font-size, font-weight covers most quick-fix needs. Should we include display, flex properties, gap? Probably not for v1 — the Source tab is the escape hatch.

> **Design system integration:** Should the Properties tab colour picker show the project's design system palette as swatches? Would help maintain consistency without the user memorising hex values.
